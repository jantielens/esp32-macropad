Binding Syntax

Labels can contain binding tokens that display live data. Static text around tokens is preserved.

Pipe Fallback

[scheme:params|fallback]

Append |value to any binding to show a custom fallback instead of ---. Useful during boot (before MQTT connects), after reconnects, and for color bindings where --- isn't a valid color.

[mqtt:solar/power;watts|0] Shows 0 until MQTT data arrives
[mqtt:light/color;hex|#ffffff] Default color until value arrives
[health:cpu|--] Custom placeholder for health data

MQTT Binding

[mqtt:topic;path;format]
topicMQTT topic to subscribe to (required)
pathJSON key to extract, dot-separated for nested. Omit or use . for raw value.
formatPrintf format string (%d, %.1f, %s). Omit for raw value.
[mqtt:sensors/temp] Raw value from topic
[mqtt:sensors/state;temperature;%.1f°C] Extract JSON key, format as float
[mqtt:home/energy;;%d W] Raw value with format (empty path = raw)
Power: [mqtt:grid;power;%d] W Static prefix & suffix around binding
[mqtt:a;temp;%.0f]° / [mqtt:b;hum;%.0f]% Multiple bindings in one label

Device Health Binding

[health:key;format]
keycpu, rssi, uptime, heap_free, heap_min, heap_largest, heap_internal, psram_free, psram_min, psram_largest, ip, hostname
formatOptional printf format string. Omit for raw value.
CPU: [health:cpu]% Device CPU usage with static suffix
[health:heap_free;%d B] Free heap with format string
[health:rssi] dBm WiFi signal strength

Date & Time Binding

[time:format;timezone]
formatstrftime format (required): %H:%M (time), %d/%m (date), %a (weekday), %Y-%m-%d, etc.
Sub-second: %ms (000-999 millis), %cs (00-99 centisec), %ds (0-9 decisec), %ums (uptime ms, standalone only)
timezoneOlson name or POSIX TZ. Omit for UTC. Examples: Europe/Amsterdam, America/New_York, Asia/Tokyo
[time:%H:%M;Europe/Amsterdam] Local time (CET/CEST)
[time:%d/%m/%Y;Europe/Amsterdam] Date as 05/03/2026
[time:%A, %B %d, %Y;Europe/Amsterdam] Thursday, March 05, 2026
[time:%a %d %b;America/New_York] Weekday and date in New York
[time:%H:%M] Current time in UTC
[time:%H:%M:%S.%ms] Time with milliseconds
[time:%ums] Device uptime in ms (no NTP needed)

Expression Binding

[expr:expression;format]

Evaluate math, comparisons, and conditionals on live binding values. Inner bindings ([mqtt:...], [health:...], etc.) are resolved first, then the expression is evaluated.

expressionMath expression with inner bindings. Supports + - * / %, comparisons > < >= <= == !=, ternary ? :, parentheses, and "quoted strings".
formatOptional printf format string after ; separator. Omit for automatic formatting.

Basic — unit conversion

[expr:[health:heap_free] / 1024;%.0f] Bytes → KB (e.g. 183)
[expr:[health:psram_free] / (1024 * 1024);%.2f] Bytes → MB (e.g. 4.00)
[expr:[mqtt:sensors/temp;value] * 9/5 + 32;%.1f] °C → °F (e.g. 72.5)

Conditional text

[expr:[mqtt:sensor/temp;value] > 30 ? "Hot" : "OK"] Threshold check → text
[expr:[health:cpu] > 80 ? "High" : "Normal"] CPU load warning

Advanced — combine multiple bindings

[expr:[mqtt:solar/power;w] - [mqtt:grid/power;w];%.0f] Net power (solar − grid)
[expr:[mqtt:solar/power;w] > 0 ? "Producing" : "Idle"] Status based on solar output

Mixed — static text + expression

Heap: [expr:[health:heap_free]/1024;%.0f] KB Static text wrapping expr
[expr:[mqtt:solar;w]/1000;%.1f]kW / [expr:[mqtt:grid;w]/1000;%.1f]kW Two expressions in one label

Sub-second timing — fast color cycling

Use modulo (%) with < to split a time range into two halves. %ms (0-999) works for periods that divide evenly into 1000. %ums (uptime ms) works for any period.

[expr:[time:%ds]%2<1?"#ff0000":"#00ff00"] Toggle every 100ms (deciseconds)
[expr:[time:%ms]%500<250?"#ff0000":"#00ff00"] Toggle every 250ms
[expr:[time:%ums]%1000<500?"#ff0000":"#00ff00"] Toggle every 500ms (any period)

Conditional color cycling — MQTT threshold warning

Combine an MQTT threshold check with fast color cycling: flash when a value exceeds a limit, otherwise use a static color.

[expr:[mqtt:energy/power]>2.5?([time:%ms]%500<250?"#ff0000":"#ffffff"):"#000000"] Flash red/white when power > 2.5, else black

Widget Data Binding

The Data Binding field in widget settings uses the same binding syntax. The resolved value is parsed as a number by the widget. Omit the format parameter — formatting would break numeric parsing.

[mqtt:home/energy/solar;power] MQTT JSON value → bar chart
[mqtt:sensors/temp] Raw MQTT value → bar chart
[health:cpu] Device CPU usage → bar chart
[expr:[mqtt:solar;w] - [mqtt:grid;w]] Computed value → bar chart
